Feature: Add a task and see it
  The user opens the app, types "+ buy milk", presses Enter, and the

  Background:
    Given the app is running with a fresh database

  Scenario: add a root task via the + prefix
    When I type "+ buy milk" in the search box
    And I press Enter
    Then the tree should contain a task titled "buy milk"
    And the task's status should be "todo"

    Then the "buy milk" row should be rendered as done

Step definitions live under packages/tests/step_definitions/; support/ spins up a Bun server with a temp ANYWHEN_STATE_DIR and a Playwright page. Same shape as kolu-master/packages/tests/.

Design rules that shaped this

  • One search box, one parser, two intents. UI branches on Input.kind, not on results.length — intent is declared, not inferred.
  • Tree is derived from a flat Task[]. No children field on Task; no completed: boolean alongside status.
  • matches (predicate) and visibleSet (ancestor closure) stay separate functions — closure is reused for any predicate.
  • parse + match + visibleSet live in one search/query.ts — they change in lockstep with the query language (Lowy: one volatility axis, one module).
  • Domain Task in shared/; SQLite row shape never leaves storage/. Swapping the store later is a one-module change.
  • Wire is @kolu/surface's Collection<Id, Task>: typed, push-based, no hand-rolled REST or polling.
  • Two Nix wrappers, kolu-style: anywhenBin (no state dir, tests crash loud if unset) + default (injects ANYWHEN_STATE_DIR). Skip kolu's fonts pkg and commit-hash stamping — neither earns its keep at this scale.
  • No top-level "Inbox / Work / Done" sections. The hierarchy is the structure; top-level tasks act as projects.
