# anywhen volatilities

Project-declared areas of volatility used by the `lowy` review — read by `lowy`
from this file (`.agency/lowy.md`) when it runs. The table lists axes that
have already shifted in this codebase or have a documented roadmap. Each
row pairs the axis with the seam where its change should land.

Rows are not findings — `/lowy` re-applies its bar (what + why + risk ×
likelihood × effect) and audits whether the boundaries under review actually
encapsulate these, rather than leaking them into consumers.

## Areas of Volatility

| Area of volatility           | What changes                                                                                                                                                                                                                                       | Why volatile (likelihood × effect)                                                                                                                                                                                                                                       | Expected encapsulation                                                                                                                                                                                                                                                            |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Filter atom grammar          | The set of atom kinds (`done:yes/no/fresh/stale`, free text, `not`), the parser, the evaluator, and the staleness clock window. README's "Further filter atoms" roadmap (`tag`, `has:body`, `root`, `under:title`) names what's next.              | High likelihood: filter atoms are an open extension point named on the roadmap. Effect: every consumer of filter results (row dimming, highlight, lever, Add-disabled gate) reads through the pipeline — a leaky grammar would touch each.                               | Grammar primitives in `packages/app/src/shared/query.ts` (one variant + one parser branch + one evaluator branch per kind); pipeline state in `packages/app/src/client/useFilter.ts` (returns `{ activeQuery, atomList, highlightQuery, rows, leverOn, toggleLever, canCreate }`). |
| Pointer / touch drag         | The drag state machine (`idle / pending / active`), long-press threshold, drop-zone math, hit-testing, mobile/stylus tuning. Recent commits churned the grip handle and long-press path.                                                          | High likelihood and effect: drag is the most-tuned input surface (touch fidelity, accessibility, mobile affordances). A change to one zone heuristic must not touch the cycle-rejection or the snapshot logic.                                                            | `packages/app/src/client/useDrag.ts` owns `DragState` (discriminated union) and the four pointer handlers. `byParentMap` / `descendantIds` live in `shared/tree.ts` because the graph walk is reused server-side. Drop-zone thresholds (`ZONE_*_RATIO`, `DRAG_LONGPRESS_MS`) in `shared/schemas.ts` so e2e tests use the same constants. |
| Inline title editing         | Begin / commit / cancel lifecycle, draft buffer, stale-commit guards (collection deltas arrive across the await; the editor must not commit into a different row's session). Multi-line bodies were the most recent feature here.                  | Real and recent: per-row inline edit is one of two write paths the user touches every session (the other is search-box Add). New affordances (rich text, validation modes, slash-commands) would land here.                                                              | `packages/app/src/client/useEdit.ts` owns the `editing` signal and the four-function lifecycle. Parent passes `onEditClosed: (id) => void` — a plain callback, never a raw `Setter` — so the parent's reactivity primitive doesn't leak into edit-management.                       |
| Per-row presentation         | The row JSX itself (drag handle, status check, title + highlight segments, body-expand toggle, edit/delete buttons), per-row memos (firstLine/body split, drop-zone match), per-row effects (focus delivery, edit-input autoselect).               | Highest churn: most user-visible commits touch this surface (status cycle, multi-line body, drag handle, dimmed class). A growing tree of conditional classes belongs to one place.                                                                                       | `packages/app/src/client/TaskRow.tsx` owns the row JSX and its row-scoped memos / effects. Receives a narrow prop bag (row data + accessors + composable returns + small callbacks); does not own any signals itself.                                                              |
| RPC error policy             | "When does a successful call clear the error toast?", "When does a destructive write require confirmation?", retry/backoff, debounced surfaces.                                                                                                    | Medium likelihood: a `confirm()` modal upgrade or a retry budget would change every call site if the policy lived inline. Effect: every mutation and every read currently routes through `callWrite` / `callQuery`, so a centralised policy is the difference between one change and N. | `packages/app/src/client/rpc.ts` exposes `createRpc(setError) → { callWrite, callQuery }` plus `confirmDestructive`. The setter is the only reactive primitive crossing the boundary; everything else is plain async functions.                                                  |
| Cache / wire fan-out         | The invariant that every single-row write to SQL must produce a Collection upsert in the same operation, otherwise the WebSocket stream and the SQL truth desync silently.                                                                          | High effect, low likelihood-per-change but easy to forget: the framework's Collection deps are synchronous and can't `await`, so the bridge is structural rather than handler-coded. A forgotten upsert is a wire-desync bug with no compile-time signal.                | `writeAndUpsert(ctx, op)` in `packages/app/src/server/router.ts` wraps both halves into one helper. Multi-row mutations (`remove` cascade, `import` wipe-and-replace, `__test__reset`) own their own fan-out because their post-state isn't "one new task value".                  |
| Task position semantics      | The float-midpoint allocation strategy, the cycle-rejection rule on `move`, the `POSITION_GAP` constant, the parent-pointer walk.                                                                                                                 | Low likelihood near-term (no LexoRank issue open; precision budget is documented at ~50 interpolations between siblings before rebalance). High effect on hit: a switch to LexoRank or a doubly-linked list would touch every move call site.                            | `taskStore.move` in `packages/app/src/storage/tasks.ts` owns position resolution. Drop semantics (`before` / `after` / `inside`) cross the wire as `MoveTarget` in `shared/schemas.ts` so the gap-allocation strategy never leaks to clients.                                       |
| Schema evolution             | Adding columns to `tasks` (body, tags, due, recurrence per the README scaffold), changing invariants (the `completedAt` ⊃ `done` constraint), changing input shapes (new procedures, widened payloads).                                            | Medium likelihood: the roadmap names tags + body + due as future fields. Effect: a schema change without the Zod `refine` would let malformed wire payloads through to SQL.                                                                                              | `packages/app/src/shared/schemas.ts` is the single source of truth; runtime types are `z.infer<typeof S>`. The wire contract in `shared/surface.ts` references these schemas directly. SQL column shape mirrored in `storage/schema.ts`. New migrations live in `storage/migrations/`.|

## Axes that intentionally stay inline today

Two axes remain inside `App.tsx` even though they qualify as their own
volatility zones. Both are below the `view-component-owns-one-axis` rule's
≥3-axis trigger and have been stable across recent commits; extracting them
today would be bureaucracy rather than encapsulation.

| Axis                       | When to extract                                                                                                                                                                                                                                                  |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Keyboard binding map (`ROW_KEY_ACTIONS` + `applyVimKey` + `handleGlobalSearch` / `handleGlobalVim`)                                                                                                                                                              | Context-conditional bindings arrive (e.g. selection-mode vs. drag-mode require different chords), or the map starts branching on row state. The dispatch grew global window-level fallback in #35 without needing extraction — that's the proof one more axis-touch is still inside the budget. |
| Backup flow (`exportTasks` / `handleImportFile` / `handleImportChange`)                                                                                                                                                                                          | Client-side cloud sync, schema-version migration on import, or a multi-step restore wizard. Today the client side is a thin pass-through to `api.export()` / `api.import()` plus a Zod safeParse; the durable rolling backup lives server-side in `storage/backup.ts`.                  |
